Q8:
How can SBTCVM and its utilities preform balanced ternary integer mathematics?

 A:
SBTCVM uses a instruction level tactic towards its VM design. meaning it interprets the VM program code using high level logic, Thus avoiding slow, complex, circuit emulation.

That's why SBTCVM is able to get such high speed operation. its mimicking the functionality of a mythical balanced ternary microprocessor and accompanying computer that doesn't even exist as a circuit diagram at this point.

of course SBTCVM needs some special support code to do that. and one of the key libraries, in fact arguably the most important, libbaltcalc.

libbaltcalc of course, works by using high-speed balanced ternary to decimal conversion, and decimal to balanced ternary conversion, to preform mathematical operations in decimal, thus using python's existing mathematics routines to quickly preform operations. those conversion routines are also used quite heavily by SBTCVM.

Q12:
Is there a frame skip setting?
 TTY rendering
Well, if you use the command shell (MK2-CS.py) alot, you might consider turning off the TTY rendering (its kinda slow).
in USERBOOT.CFG (its created on first-run of the VM in VMUSER)
find the line: "#ttystyle=0" and set it to "ttystyle=1"

CPU Wait time

The amount of time per cycle to sleep in seconds (as a floating point number)

You may try, if your computer is fast enough, shortening the cpu delay.
find the line "#CPUWAIT=(0.0005)" and uncomment it, and try changing the time to a shorter time

likewise, if SBTCVM is using too much CPU for your taste, try making this time longer.

Frame Skipping

number of clock cycles to wait between display updates.

also you can try setting fskip (find the line "#fskip=1") to a higher number. by default fskip is set to 1, that means that every other clock cycle's frame is rendered. if you have say an average operations/second of 2000, you might try fskip=100, as with 2000 operations/second, by default the display is being updated 1000 times a second.  so yea, fskip is worth adjusting a bit.

notes:
  • The data readouts are disabled by default for speed reasons. toggle them with F4.
  • if your debugging a trom, try F2, this helps step through a program manually by pressing return, though the keyscan system doesn't work in this mode. press F2 again to return to normal execution.
  • using MK2-RUN.py's -tos option will give you an approx operations/second tally. note that execution logging (MK2-RUN.py -le) also reports this in the resulting log file, but it does slow down the VM somewhat from normal due to the per-cycle status reporting to the log.
